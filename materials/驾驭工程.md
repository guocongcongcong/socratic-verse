# 驾驭工程

## 这才是真正的门槛

- **Prompt Engineering** → 让 Agent 听懂你
- **Context Engineering** → 让 Agent 看到该看的
- **Harness Engineering** → 让 Agent 持续、可靠地完成任务

前两者是"一次对话"层面的优化。Harness 是"系统"层面的工程——你不再是写一个 prompt，而是在设计一个能自主运行的**系统**。

---

## 六层架构

```
6. 约束校验 & 恢复  ← 权限控制、Hook 拦截、沙盒
5. 评估 & 观测      ← 追踪每一步决策，事后审计
4. 状态 & 记忆      ← 跨 session 持久化（Memory/Skill/CLAUDE.md）
3. 执行编排         ← Agent Loop 设计，何时结束、何时重试
2. 工具系统         ← 工具注册、调用、错误处理
1. 上下文管理       ← 按需注入、压缩、切换
```

---

## 逐层拆解

### 第1层：上下文管理

你已经熟悉了。就是上一章 [[上下文工程]] 的内容。

**Hermes 中的实际例子：** 在当前 session 中，上下文管理通过多层注入实现——`system-reminder` 标签注入当前日期、可用 Skills 列表、Memory 索引（MEMORY.md）；CLAUDE.md 作为项目级上下文在会话启动时加载；Skills 通过 `Skill` 工具按需加载（如 `superpowers:brainstorming`），避免一次性塞入所有内容。对应的核心机制：系统提示词中的 `<system-reminder>` 注入块 + Memory 目录的文件索引 + Skill 的懒加载架构。压缩策略则在上下文接近窗口限制时自动触发（对应《马书》第9-12章的分析）。

### 第2层：工具系统

不只是"有哪些工具"，还包括：
- 工具发现（Agent 怎么知道有哪些工具可用）
- 工具注册（加一个新工具要改多少代码）
- 错误处理（工具挂了 Agent 怎么办）
- 权限控制（Agent 能用哪些工具，不能用哪些）

**Hermes 中的实际例子：** 当前可用的工具集包括 Bash、Read、Write、Edit、Glob、Grep、Agent、TaskCreate、Skill、WebFetch、WebSearch 等 20+ 个工具。每个工具有明确的参数 schema，Agent 通过系统提示词中的工具列表获知可用工具。工具分为两类——只读工具（Read、Glob、Grep）自动通过权限检查，写工具（Write、Edit、Bash）需要用户确认。对应《马书》第2章：Hermes 的工具注册虽然不像 CC 那样有 40+ 工具和 `buildTool()` 工厂函数，但在"失败关闭默认值"（写操作默认需确认）和"输入感知的属性"（同一 Bash，ls 和 rm 权限不同）上遵循相同的设计哲学。

### 第3层：执行编排

这是 Agent Loop 的工程化：

```
while 任务未完成 and 步数 < 上限:
    Thought → 选择工具 → 调用 → 观察结果 → 判断
```

关键点：
- **步数上限**：防止 Agent 无限循环（Hermes 默认 100 步）
- **超时机制**：单个工具调用超过 N 秒就中断
- **重试策略**：工具调用失败后，给 Agent 机会重试

**Hermes 中的实际例子：** Hermes 的 Agent Loop 是一个标准的 ReAct 循环——每轮迭代中，Agent 产生思考（thinking）、选择工具、等待结果、判断是否继续。步数上限通过 `maxTurns` 控制（对应系统提示词中的步数限制）。工具调用超时在 Bash 工具上有 `timeout` 参数（默认 120000ms，最大 600000ms）。重试策略体现为：当工具调用失败（如权限被拒），Agent 会收到错误消息并被给予调整策略的机会。对比《马书》第3章的 CC queryLoop——CC 的循环远比 ReAct 复杂（7 种 Continue 转换 + 10 种 Terminal 终止原因），包含自动压缩、模型降级、输出截断恢复等生产级机制，而 Hermes 的设计更轻量但在核心原理上一脉相承。

### 第4层：状态 & 记忆

Agent 不能每次对话都"失忆"。需要：
- **Memory**：跨会话的碎片信息
- **Skill**：按需加载的操作流程
- **CLAUDE.md**：项目级上下文

**Hermes 中的实际例子：** Memory 系统在 `/memory/` 目录下，分为四种类型——user（用户偏好/角色）、feedback（行为指导）、project（项目背景）、reference（外部系统指针）。每种类型有独立的文件，通过 MEMORY.md 索引加载。Skill 系统分为 superpowers 技能（如 brainstorming、TDD、debugging）和 gstack 技能（如 qa、design-review、browse），通过 `Skill` 工具按需激活。CLAUDE.md 在项目根目录定义项目级规范和约束。这个组合对应《马书》第19章（CLAUDE.md 作为覆盖层）、第22章（技能系统）、第24章（跨会话记忆）。

### 第5层：评估 & 观测

你没法改进你看不到的东西。需要：
- 记录 Agent 每一步做了什么（类似日志）
- 事后审查 Agent 的决策链路
- 找到"Agent 为什么做错了"的根因

**具体实现方式：**

1. **任务追踪（Task List）**：通过 TaskCreate / TaskList / TaskUpdate 工具，将复杂任务分解为可追踪的子任务。每个任务有明确的状态（pending → in_progress → completed），相当于 Agent 给自己建立了一张"工作看板"。任务完成后可通过 TaskGet 回溯执行记录。

2. **对话日志（Conversation Log）**：每次会话的完整消息流就是天然的审计日志——每一轮 Thought → Tool Call → Tool Result 都被记录。事后可以复盘"Agent 在第 N 步做了什么决策，依据是什么"。

3. **产出验证（Output Verification）**：通过 `superpowers:verification-before-completion` 技能，要求 Agent 在声称完成任务前实际运行验证命令并确认输出。这是一层"自检"机制——不是靠 Agent 自己说做完了，而是靠 Agent 拿出证据。

4. **Cron 监控（Scheduled Checks）**：通过 CronCreate 工具设置定时检查，如每隔 N 分钟检查部署状态、服务健康等，实现持续的被动观测。

**Hermes 中的实际例子：** Hermes 的观测体系是分散但可组合的——TaskCreate 提供过程可见性，对话历史提供完整审计轨迹，verification-before-completion 技能提供结果可信度保证，Cron 提供持续监控窗口。对比《马书》第29章的可观测性工程：CC 有更体系化的 `logEvent` 遥测系统，而 Hermes 的观测更依赖"让 Agent 自己记录和验证"——两种模式各有适用场景。

5. **benchmark 技能**：Hermes 的 `benchmark` 技能提供了性能回归检测能力——建立页面加载时间、Core Web Vitals 和资源大小的基线，比较 PR 前后变化，追踪性能趋势。这对应《马书》中的 A/B 测试思维（第7章）。

6. **health 技能**：代码质量仪表盘——包装类型检查器、linter、测试运行器、死代码检测器，计算加权复合评分（0-10），追踪趋势。这是一种"把观测嵌入到 Harness 中"的做法——不是手动检查，而是让系统自动给出健康评分。

### 第6层：约束 & 恢复

Agent 会犯错。你需要：
- **Hook**：拦截特定事件（如"Agent 要删文件了"）
- **沙盒**：限制 Agent 的操作范围
- **恢复机制**：出错后自动回滚或人工介入

**具体实现方式：**

1. **权限系统（Permission System）**：工具调用前经过权限检查。写操作（Write、Edit、Bash 中的危险命令）需要用户确认。`settings.json` 中的 `permissions` 字段定义允许/拒绝的工具和路径。这是一个"人在环中"（human-in-the-loop）的安全闸门。

2. **Hook 拦截**：系统提示词中定义了 Hook 机制——用户可配置在特定事件触发时执行的 shell 命令。例如，在每次用户提交消息前运行的 `<user-prompt-submit-hook>`。Hook 可以返回阻塞错误，阻止 Agent 继续执行。对应《马书》第18章（Hooks 系统）。

3. **目录冻结（Freeze/Guard）**：Hermes 的 `freeze` 技能限制文件编辑范围到指定目录——阻止 Edit 和 Write 操作超出允许路径。`guard` 技能组合了目录冻结和 `careful` 的危险命令警告（rm -rf、DROP TABLE、force-push 等被拦截）。对应《马书》第18b章（沙箱系统）。

4. **恢复机制**：
   - **步数上限恢复**：Agent 达到 maxTurns 后，用户可要求 Agent 总结已完成工作和待办事项，在新会话中继续
   - **上下文压缩恢复**：当上下文接近窗口限制，系统自动压缩旧消息，保留关键信息
   - **Error Escalation**：工具调用失败时，错误消息被传递给 Agent，Agent 可以尝试替代方案
   - **Session 恢复**：通过 `context-save` / `context-restore` 技能，在会话中断前保存状态（git 状态、决策记录、剩余工作），新会话中恢复

5. **安全审计（CSO）**：`cso` 技能提供基础设施优先的安全审计——密钥考古、依赖供应链、CI/CD 管道安全、LLM/AI 安全、技能供应链扫描、OWASP Top 10、STRIDE 威胁建模。这对应《马书》第17b章（提示注入防御）的安全纵深防御思维。

**Hermes 中的实际例子：** Hermes 的约束体系是"多层防护"——权限系统是第一道门（阻止未授权操作），Hook 是第二道门（用户自定义拦截逻辑），freeze/guard 是第三道门（空间限制），步数/超时限制是最后的安全网（防止失控）。对比《马书》第16章的 CC 权限系统——CC 有更细粒度的 `alwaysAllow`/`alwaysAsk`/`alwaysDeny` 三级权限，加上 YOLO 分类器（第17章）自动判断操作安全性。Hermes 的设计更依赖人在环中确认，CC 更倾向于自动化分类。

## Harness 的反面——没有 Harness 的 Agent

想象一个没有 Harness 的"裸 Agent"：你把 API key 给模型，给它一个 Bash 工具，然后说"帮我重构这个项目"。以下是会发生的事：

### 问题1：无限循环

Agent 没有步数上限，没有超时机制。它可能在第 50 轮反复尝试同一个失败的工具调用——每次失败都收到类似的错误，每次都给出类似的"我来换一种方式"的承诺。因为没有 `maxTurns` 和 timeout 来打断它，你需要手动 Ctrl+C。

**有 Harness 之后：** `maxTurns=100`，单工具 `timeout=120s`，单轮 token 预算——三条硬线让失控自动终止。

### 问题2：权限失控

Agent 为了"解决"一个 bug，可能会：
- 删除你以为是临时文件的 `~/.zshrc`
- 往 `/etc/hosts` 里写测试用的域名映射
- `npm install` 一个有漏洞的依赖
- `git push --force` 到 main 分支

没有权限系统，Agent 每一句"我帮你修好了"都可能是在埋一颗炸弹。

**有 Harness 之后：** 权限系统在工具调用前拦截，写操作需要用户确认。freeze 限制操作目录，careful 模式对危险命令额外警告。

### 问题3：记忆丢失

Agent 工作了 30 轮，分析了 15 个文件，改了 8 个文件，还有 3 个待办项。然后上下文窗口满了——或者会话断了重连——Agent 变成一张白纸。你重新描述需求，Agent 从零开始。

**有 Harness 之后：** Memory 跨会话持久化关键信息。context-save/restore 保存和恢复工作状态。Task List 记录进度——即使会话中断，新会话读取 Task List 即可接续。

### 问题4：无法调试

Agent 做了 88 步操作后给出了一个错误结果。你想知道：
- 它在第几步开始跑偏的？
- 哪次工具调用返回了异常结果？
- 为什么它选择策略 A 而不是策略 B？

没有观测系统，你只能对着最终结果和 88 步的混沌感到沮丧。"到底是模型推理错了还是工具返回错了？"——不知道。

**有 Harness 之后：** 对话日志提供完整审计轨迹。Task List 标记每步进行状态。verification-before-completion 要求在声称完成前提供验证证据。benchmark 和 health 技能持续追踪指标变化。

### 本质区别

```
裸 Agent：    Prompt → API → 工具 → 结果
有 Harness：  上下文注入 → 权限检查 → Agent Loop(步数限制+超时+重试) → 工具执行 → 结果观测 → 记忆持久化 → Hook 校验 → 下一轮
```

裸 Agent 是一个"调用"。有 Harness 的 Agent 是一个"系统"。

## Hermes 的 Harness 架构一览

从用户视角看，Hermes 的 Harness 是如何把一个个工具调用编织成可靠工作流的：

```
                        ┌──────────────────┐
                        │   用户输入任务     │
                        └────────┬─────────┘
                                 │
                                 ▼
                        ┌──────────────────┐
                        │  1. Memory 加载   │
                        │  MEMORY.md 索引   │
                        │  + 相关 memory 文件│
                        └────────┬─────────┘
                                 │
                                 ▼
                        ┌──────────────────┐
                        │  2. 上下文注入     │
                        │  CLAUDE.md       │
                        │  system-reminder │
                        │  Skill 列表       │
                        └────────┬─────────┘
                                 │
                                 ▼
           ┌─────────────────────────────────────┐
           │         3. Agent Loop               │
           │                                     │
           │  ┌───────────────────────────────┐  │
           │  │ Thought：理解任务，拆解步骤     │  │
           │  └───────────┬───────────────────┘  │
           │              │                      │
           │              ▼                      │
           │  ┌───────────────────────────────┐  │
           │  │ Skill 激活：按需加载领域 Skill  │  │
           │  │ brainstorming / TDD / debug   │  │
           │  └───────────┬───────────────────┘  │
           │              │                      │
           │              ▼                      │
           │  ┌───────────────────────────────┐  │
           │  │ 工具选择：从 20+ 工具中选最合适的│  │
           │  └───────────┬───────────────────┘  │
           │              │                      │
           │              ▼                      │
           │  ┌───────────────────────────────┐  │
           │  │ 权限检查：只读工具自动通过      │  │
           │  │ 写操作等待用户确认             │  │
           │  └───────────┬───────────────────┘  │
           │              │                      │
           │              ▼                      │
           │  ┌───────────────────────────────┐  │
           │  │ 工具执行 + 结果观测            │  │
           │  │ Task 状态更新                 │  │
           │  └───────────┬───────────────────┘  │
           │              │                      │
           │              ▼                      │
           │  ┌───────────────────────────────┐  │
           │  │ 判断：任务完成了？              │  │
           │  │ NO → 回到 Thought（步数+1）    │  │
           │  │ YES → 进入验证阶段             │  │
           │  └───────────────────────────────┘  │
           └─────────────────────────────────────┘
                                 │
                                 ▼
                        ┌──────────────────┐
                        │  4. Hook 拦截     │
                        │  pre-completion  │
                        │  stop hook 检查   │
                        └────────┬─────────┘
                                 │
                                 ▼
                        ┌──────────────────┐
                        │  5. 记忆持久化    │
                        │  更新 Memory     │
                        │  可 context-save │
                        └────────┬─────────┘
                                 │
                                 ▼
                        ┌──────────────────┐
                        │  结果返回用户     │
                        └──────────────────┘
```

**五个关键环节：**

1. **Memory →** 会话开始时加载 MEMORY.md 索引，将用户偏好、项目背景、行为指导注入上下文。这确保 Agent 不是"一张白纸"——它知道你之前告诉过它什么。

2. **Skill →** Agent 根据任务类型按需激活 Skill。Skill 不是一次性全部加载（那样会浪费 token），而是在 Agent 识别到适用场景时通过 `Skill` 工具激活。Skill 的内容直接注入系统提示词，重写 Agent 的默认行为。

3. **Tool →** Agent 从 20+ 工具中选择最合适的。只读工具自动执行，写操作触达用户确认。工具执行是循环的核心——每一步的结果反馈回模型，影响下一步决策。

4. **Agent Loop →** 这是 Harness 的"心脏"。每一次迭代都是：判断状态 → 选择动作 → 执行 → 观察 → 更新状态。步数上限、超时、重试策略都在这一层。循环终止条件包括任务完成（模型不再调用工具）、达到步数上限、用户中断。

5. **Hook →** 在关键截点（用户提交消息前、Agent 声称完成时）触发检查。Hook 可以返回阻塞错误让 Agent 修正，也可以静默记录日志。这是用户扩展 Harness 行为的主要方式。

对比一个"裸 API 调用"——用户消息进来，模型回复出去——Hermes 的 Harness 在每个环节都加入了安全网：Memory 防止失忆，Skill 提供领域知识，权限防止事故，Loop 防止失控，Hook 提供可扩展性。
