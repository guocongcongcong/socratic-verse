在过去的一年中，我们与数十个团队合作，在各行各业构建大语言模型（LLM）Agent。我们发现，最成功的实现并非使用了复杂的框架或专门的库，而是基于简单、可组合的模式构建的。

本文中，我们将分享从客户合作和自身构建 Agent 的过程中积累的经验，并为开发者提供构建高效 Agent 的实用建议。

## 什么是 Agent？

"Agent"可以有多重定义。一些客户将 Agent 定义为完全自主的系统，能够在较长时间内独立运行，使用各种工具完成复杂任务。另一些客户则用该术语来描述遵循预定义工作流的更具规范性的实现。在 Anthropic，我们将所有这些变体统称为 agentic systems（智能体系统），但在 Workflow（工作流）和 Agent（智能体）之间做出了重要的架构区分：

- **Workflow（工作流）** 是通过预定义的代码路径来编排 LLM 和工具的系统。
- **Agent（智能体）** 则是 LLM 动态地指导自身流程和工具使用的系统，它们自主掌控如何完成任务。

下文将详细探讨这两种 agentic systems。在附录一（"Agent 实战"）中，我们将描述客户在应用这类系统时发现特别有价值的两个领域。

## 何时使用（以及不使用）Agent

在使用 LLM 构建应用时，我们建议寻找最简单的解决方案，仅在必要时才增加复杂性。这意味着可能根本不需要构建 agentic systems。Agentic systems 通常以延迟和成本为代价来换取更好的任务表现，你需要权衡这种取舍是否合理。

当确实需要更高的复杂性时，Workflow 为定义明确的任务提供了可预测性和一致性，而在需要灵活性和模型驱动决策的大规模场景中，Agent 则是更好的选择。然而，对于许多应用而言，通过检索和上下文示例来优化单次 LLM 调用通常就足够了。

## 何时以及如何使用框架

有许多框架可以让 agentic systems 的实现更加容易，包括：

- Claude Agent SDK
- AWS 的 Strands Agents SDK
- Rivet，一个拖拽式 GUI LLM 工作流构建器
- Vellum，另一个用于构建和测试复杂工作流的 GUI 工具

这些框架通过简化标准底层任务（如调用 LLM、定义和解析工具、链式调用等）降低了入门门槛。然而，它们往往创建了额外的抽象层，可能掩盖底层的 prompt 和响应，使调试变得更加困难。它们还可能诱使开发者在简单的设置就足够时添加不必要的复杂性。

我们建议开发者从直接使用 LLM API 开始：许多模式只需几行代码即可实现。如果你确实使用了某个框架，请确保理解其底层代码。对底层机制的误解是客户出错的常见原因。

参见我们的 cookbook 获取一些示例实现。

## 构建块、工作流与 Agent

在本节中，我们将探讨在生产环境中观察到的 agentic systems 的常见模式。我们将从基础的构建块——augmented LLM（增强型 LLM）——开始，逐步增加复杂性，从简单的组合式工作流到自主 Agent。

### 构建块：增强型 LLM

Agentic systems 的基本构建块是一个经过增强的 LLM，具备检索、工具和记忆等增强能力。我们当前的模型可以主动使用这些能力——生成自己的搜索查询、选择合适的工具，并确定要保留哪些信息。

我们建议关注实现的两个关键方面：为你的特定用例定制这些能力，以及确保它们为你的 LLM 提供简单、文档完善的接口。虽然实现这些增强有很多方式，但一种方法是使用我们最近发布的 Model Context Protocol（MCP，模型上下文协议），它允许开发者通过简单的客户端实现集成日益壮大的第三方工具生态。

在下文中，我们将假定每次 LLM 调用都可以访问这些增强能力。

### Workflow：Prompt Chaining（提示链）

Prompt Chaining 将任务分解为一系列步骤，每个 LLM 调用处理前一个调用的输出。你可以在任何中间步骤添加程序化检查（见下图中的 "gate"（关卡）），以确保流程仍在正轨上。

**何时使用此工作流：** 此工作流适用于任务可以轻松、清晰地分解为固定子任务的场景。其主要目标是通过让每次 LLM 调用完成一个更简单的任务，以延迟换取更高的准确性。

**Prompt Chaining 有用的示例：**
- 生成营销文案，然后将其翻译成另一种语言。
- 撰写文档大纲，检查大纲是否符合特定标准，然后基于大纲撰写文档。

### Workflow：Routing（路由）

Routing 对输入进行分类，并将其引导到专门的后继任务中。此工作流实现了关注点分离，并允许构建更专业的 prompt。如果不使用此工作流，针对某一类输入的优化可能会损害其他输入的性能。

**何时使用此工作流：** Routing 适用于存在明显不同类别、且各类别最好分别处理的复杂任务，同时分类可以通过 LLM 或更传统的分类模型/算法准确处理。

**Routing 有用的示例：**
- 将不同类型的客户服务查询（一般问题、退款请求、技术支持）引导到不同的下游流程、prompt 和工具中。
- 将简单/常见问题路由到更小、更具成本效益的模型（如 Claude Haiku），将困难/不寻常的问题路由到能力更强的模型（如 Claude Sonnet），以优化整体性能。

### Workflow：Parallelization（并行化）

LLM 有时可以同时处理一个任务的不同部分，并通过程序化方式聚合其输出。此工作流 Parallelization 有两种关键变体：

- **Sectioning（分段）：** 将任务分解为独立运行的并行子任务。
- **Voting（投票）：** 多次运行同一任务以获取多样化的输出。

**何时使用此工作流：** 当分解后的子任务可以并行执行以提升速度，或需要多个视角或多次尝试以获得更高置信度的结果时，Parallelization 非常有效。对于涉及多个考量因素的复杂任务，当每个考量因素由单独的 LLM 调用处理时，LLM 通常表现更好，因为可以集中注意力于每个特定方面。

**Parallelization 有用的示例：**

*Sectioning（分段）：*
- 实施 guardrails（防护栏），其中一个模型实例处理用户查询，同时另一个模型实例筛查不当内容或请求。
- 自动化评估 LLM 性能的 evals（评估流程），其中每个 LLM 调用评估模型性能的不同方面。

*Voting（投票）：*
- 审查代码漏洞，多个不同的 prompt 分别审查代码并在发现问题时进行标记。
- 评估某段内容是否不当，多个 prompt 评估不同方面。

### Workflow：Orchestrator-workers（编排器-工作者）

在 Orchestrator-workers 工作流中，一个中央 LLM 动态分解任务，将其委派给 worker LLM（工作者 LLM），并综合它们的结果。

**何时使用此工作流：** 此工作流非常适合无法预先预测所需子任务的复杂任务。虽然结构与 Parallelization 相似，但关键区别在于其灵活性——子任务并非预定义的，而是由 orchestrator（编排器）根据具体输入动态决定的。

**Orchestrator-workers 有用的示例：**
- 每次需要对多个文件进行复杂修改的编码产品。
- 涉及从多个来源收集和分析信息的搜索任务。

### Workflow：Evaluator-optimizer（评估器-优化器）

在 Evaluator-optimizer 工作流中，一个 LLM 调用生成响应，另一个 LLM 调用在循环中提供评估和反馈。

**何时使用此工作流：** 当有明确的评估标准，且迭代优化能产生可衡量的价值时，此工作流特别有效。两个表明适合使用此模式的标志是：第一，当人类提出反馈时，LLM 的响应能够得到明显改善；第二，LLM 能够提供此类反馈。

**Evaluator-optimizer 有用的示例：**
- 文学翻译，其中存在翻译 LLM 最初可能无法捕捉到的细微差别，但评估 LLM 可以提供有用的批评。
- 复杂的搜索任务，需要多轮搜索和分析以收集全面信息。

## Agent（智能体）

随着 LLM 在关键能力上的成熟——理解复杂输入、进行推理和规划、可靠地使用工具、从错误中恢复——Agent 正在生产环境中崭露头角。Agent 的工作始于人类用户的命令或交互式讨论。一旦任务明确，Agent 便独立地规划和运行，可能需要返回人类获取进一步的信息或判断。在执行过程中，Agent 在每一步从环境中获取"ground truth（真实情况）"（如工具调用结果或代码执行结果）以评估进展至关重要。然后，Agent 可以在检查点或遇到障碍时暂停以获取人类反馈。任务通常在完成时终止，但通常也会设置停止条件（如最大迭代次数）以保持控制。

Agent 可以处理复杂的任务，但其实现通常很直接。它们通常只是 LLM 在循环中根据环境反馈使用工具。因此，精心设计工具集及其文档至关重要。

**何时使用 Agent：** Agent 适用于开放式问题——即难以或不可能预测所需步骤数量、且无法硬编码固定路径的场景。LLM 可能会运行很多个回合，你必须对其决策能力有一定程度的信任。Agent 的自主性使其非常适合在受信任的环境中扩展任务。

Agent 的自主特性意味着更高的成本和潜在的累积错误风险。我们建议在沙箱环境中进行广泛测试，并配合适当的 guardrails（防护措施）。

**Agent 有用的示例：**
- 一个编码 Agent 来解决 SWE-bench 任务，这些任务涉及根据任务描述对多个文件进行修改。
- 我们的 "computer use"（计算机使用）参考实现，其中 Claude 使用计算机来完成任务。

## 组合和定制这些模式

这些构建块并非教条式的规定。它们是常见的模式，开发者可以根据不同的用例进行塑造和组合。与任何 LLM 功能一样，成功的关键在于衡量性能并迭代优化实现。再强调一次：你应当只在能够明显改善结果时才增加复杂性。

## 总结

在 LLM 领域的成功不在于构建最复杂的系统，而在于构建适合你需求的系统。从简单的 prompt 开始，通过全面的评估进行优化，只在更简单的解决方案无法满足需求时才添加多步 agentic systems。

在实现 Agent 时，我们尝试遵循三个核心原则：
1. 保持 Agent 设计的简洁性。
2. 通过明确展示 Agent 的规划步骤来优先保证透明性。
3. 通过详尽的工具文档和测试来精心打造你的 agent-computer interface（ACI，智能体-计算机接口）。

框架可以帮助你快速起步，但在走向生产环境时，不要犹豫减少抽象层并使用基础组件构建。通过遵循这些原则，你可以创建出不仅强大，而且可靠、可维护并受到用户信任的 Agent。

## 附录一：Agent 实战

我们与客户的工作揭示了两个特别有前景的 AI Agent 应用领域：

### A. 客户支持

客户支持将熟悉的聊天机器人界面与通过工具集成增强的能力相结合。这是开放式 Agent 的自然适用场景，因为：
- 支持交互自然遵循对话流程，同时需要访问外部信息和执行操作
- 可以集成工具来拉取客户数据、订单历史和知识库文章
- 诸如退款或更新工单等操作可以通过程序化方式处理
- 成功可以通过用户定义的解决标准来清晰衡量

### B. 编码 Agent

软件开发领域已经展示了 LLM 功能的巨大潜力，能力从代码补全演进到自主问题解决。Agent 特别有效的原因在于：
- 代码解决方案可通过自动化测试进行验证
- Agent 可以使用测试结果作为反馈来迭代解决方案
- 问题空间定义明确且结构化
- 输出质量可以客观衡量

在我们自己的实现中，Agent 现在可以仅根据 pull request 描述解决 SWE-bench Verified 基准测试中的真实 GitHub issue。然而，虽然自动化测试有助于验证功能，但人工审查对于确保解决方案符合更广泛的系统要求仍然至关重要。

## 附录二：对你的工具进行 Prompt Engineering

无论你在构建哪种 agentic system，工具都可能成为你 Agent 的重要组成部分。工具使 Claude 能够通过在我们的 API 中定义其确切结构和规范来与外部服务和 API 交互。工具的定义和规范应当获得与你的整体 prompt 同等程度的 prompt engineering 关注。

同一操作通常有多种指定方式。例如，你可以通过编写 diff 来指定文件编辑，也可以通过重写整个文件来指定。对于结构化输出，你可以在 markdown 中返回代码，或是在 JSON 中返回。在软件工程中，这些差异是表面的。然而，有些格式对 LLM 来说比其他格式更难编写。

我们关于选择工具格式的建议如下：
1. 给模型足够的 token 来"思考"，避免它在还没想清楚之前就陷入死胡同。
2. 保持格式接近模型在互联网文本中自然见过的形式。
3. 确保没有格式"开销"，例如需要精确计数数千行代码，或对它编写的任何代码进行字符串转义。

一条经验法则是：想一想人们在 human-computer interfaces（HCI，人机接口）上投入了多少精力，然后计划投入同等的精力来创建良好的 agent-computer interfaces（ACI，智能体-计算机接口）。以下是一些如何做到这点的思路：
- 站在模型的角度思考。基于描述和参数，如何使用这个工具是显而易见的吗？
- 如何更改参数名称或描述以使事情更加清晰？
- 测试模型如何使用你的工具：输入大量示例来观察模型会犯什么错误，并迭代改进。
- 对你的工具进行 Poka-yoke（防错设计）。更改参数，使犯错变得更加困难。

在为 SWE-bench 构建我们的 Agent 时，我们实际在优化工具上花费的时间比优化整体 prompt 更多。例如，我们发现当 Agent 离开根目录后，模型在使用相对文件路径的工具时会出错。为了修复这个问题，我们将工具改为始终要求绝对文件路径——结果发现模型使用这种方法时毫无差错。
