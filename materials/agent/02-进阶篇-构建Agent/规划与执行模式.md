# 规划与执行模式

> Agent循环深度解析里提了四种循环模式，其中 ReWOO 的"先计划再执行"是 Token 效率最高的。这一篇深入拆解：计划怎么写、执行怎么并行、出错了怎么补救。

---

## 问题：ReAct 的"脱口秀式"执行

ReAct 像一个被临时拉上台的脱口秀演员——每句话都基于前一句的反应，没有机会提前想完整稿。

查 10 个网站的信息来回答一个问题：
- 步骤 1：搜索网站 A → 得到结果
- 步骤 2：分析结果，决定搜网站 B → 带着步骤 1 的全部上下文
- 步骤 3：分析结果，决定搜网站 C → 带着步骤 1 + 2 的全部上下文
- ...
- 步骤 10：带着前 9 步的全部上下文回答 → prompt 里 80% 是历史，20% 是真正有用的信息

**更糟糕的是**：步骤 2 到步骤 9 之间没有依赖关系——它们完全可以同时执行。

> 🤔 **思考题**：你让 Agent "比较 A B C 三家公司的财报"。三家公司独立，为什么 Agent 非得一家一家查？

## ReWOO：先画地图，再走路

ReWOO = **Re**asoning **W**ith**O**ut **O**bservation

核心理念：**计划的生成和执行应该分离**。

### 三阶段架构

```
阶段一：Plan（规划）
  问题 → Planner LLM → 执行计划 DAG
  "我需要：① 查 A 公司财报 ② 查 B 公司财报 ③ 查 C 公司财报 ④ 三者对比"

阶段二：Execute（执行）
  计划 DAG → Worker 并行执行 → 收集证据
  Worker1: 查 A → 结果A
  Worker2: 查 B → 结果B  ← 三个 Worker 同时跑
  Worker3: 查 C → 结果C

阶段三：Solve（求解）
  问题 + 计划 + 全部证据 → Solver LLM → 最终答案
  "根据三家公司财报，得出对比结论"
```

### 为什么 Token 效率高 ~5 倍

| | ReAct | ReWOO |
|---|-------|-------|
| Planner 调用 | 无（规划和执行混在一起） | 1 次大调用（生成完整计划） |
| Worker 调用 | 10 次（每次带全部历史） | N 次小调用（每次只带自己的任务） |
| Solver 调用 | 最后一次调用（带全部历史） | 1 次（带计划+全部证据） |
| 并行 | 不可能 | 无依赖的 Worker 可并行 |
| **Token 总量** | 高（历史重复承载） | **低（历史不重复）** |

## Plan-and-Execute：给计划加个"反悔"机会

ReWOO 的核心弱点：**计划一旦生成，就焊死了**。Worker 执行时发现"哦，我还需要再查一个东西"——Solver 看不到这个需求。

Plan-and-Execute 加了一个步骤：**Replanner**。

```
Plan → Execute → Replan（检查：执行结果够不够？）
                        ↓ 不够
                   修改计划 → Execute → Replan → ...
                        ↓ 够了
                      Solve
```

Replanner 拿到执行结果后问自己：
1. 所有计划步骤都完成了吗？
2. 结果质量够高吗？
3. 有没有新发现需要追加计划？

**代价**：多了一个 Replanner LLM 调用。但这个调用比 ReAct 的"每一步都带全部历史"便宜得多。

> 🤔 **思考题**：什么情况下 Replanner 会一直说"不够"，导致无限循环？怎么防止？

## Plan-and-Act：为超长任务设计

Erdogan et al.（ICML 2025）把 ReWOO 推向了更极端的方向：

**200+ 步的 Web/Mobile 任务**。用合成数据训练 Planner（告诉它"你是一个 Web 操作专家，面对这个页面，你需要的操作序列是…"）。

关键创新：**计划是分层的**。

```
战略层：完成"预订机票"这个大目标
  ├── 战术层：① 打开订票网站 ② 搜索航班 ③ 选座 ④ 支付
  │   ├── 操作层：②-1 输入出发地 ②-2 输入目的地 ②-3 选日期 ②-4 点击搜索
  │   └── 操作层：③-1 阅读座位图 ③-2 选靠窗座位 ③-3 确认
  └── ...
```

每一层失败时，不需要重建整个计划——只需重建失败的那一层。

## 实战选择指南

| 任务类型 | 推荐模式 | 理由 |
|----------|----------|------|
| "帮我查一下 X" | ReAct（或直接单次调用） | 不需要计划 |
| "帮我查 A B C 三个东西，然后比较" | ReWOO | 三个 Worker 并行查，一个 Solver 综合 |
| "帮我调研 X 主题" | Plan-and-Execute | 搜索结果可能引出新方向，需要重规划 |
| "帮我操作网页完成预订" | Plan-and-Act | 分层规划，每层独立失败恢复 |
| "帮我修复一个 Bug" | ReAct（带测试做评估器） | 环境不确定，调试路径不可预测 |

## 一个设计练习

设计一个"自动写周报"的 Agent：

1. 它的输入是什么？
2. 它需要收集哪些独立信息？（这些可以并行吗？）
3. 它的计划 DAG 长什么样？
4. 什么情况下需要 replan？
5. Solver 需要拿到什么才能生成好的周报？

试着画出这个 Agent 的 Plan → Execute → Solve → Replan 全流程。
