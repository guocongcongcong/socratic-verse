# 写 Skill

## 什么是 Skill

Skill 就是 Agent 的 **SOP（标准操作流程）**。你告诉 Agent："遇到这类任务时，按这个步骤做"。

它不是代码。它是 Markdown 文件。Agent 读到它，就会按它的指引行动。

---

## Skill 的最小结构

```markdown
---
name: my-first-skill
description: 当用户说 X 时触发
---

# My First Skill

## 触发条件
- 用户说 "翻译这篇文章"
- 用户提到 "多语言"

## 操作步骤
1. 先读取文件内容
2. 逐段翻译成中文
3. 保持原文格式
4. 输出到新文件

## 踩坑记录
- 代码块内的文字不要翻译
- 专有名词保持原文
```

Agent 读到这个 Skill 后，当用户说"翻译这篇文章"时，就会自动按步骤执行。

---

## 写 Skill 的五条铁律

### 1. 一个 Skill 只做一件事

✅ 翻译 Skill、搜索 Skill、格式化 Skill —— 分开写
❌ "万能工具箱 Skill" —— Agent 会困惑

**反面例子**：有人写了一个 `project-helper` Skill，触发条件是"用户需要帮助"，步骤里塞了代码审查、文档生成、部署检查。结果 Agent 每次对话都触发它，但大部分步骤跟当前任务无关，既浪费 token 又打乱执行节奏。Agent 不是人，不会"挑着用"——给它一个 Skill，它就全量执行。

### 2. 触发条件要明确

Agent 需要知道**什么时候**用它。不明确的触发条件 = 永远不会被触发。

✅ "用户说 '记到知识库' 或 '存入 wiki'"
❌ "用户需要记录东西时"

**反面例子**：一个 Skill 的触发条件写成"当用户提到代码质量相关的事情"。但"代码质量"太模糊了——用户说"这段代码好丑"、"这个函数太长"、"lint 报错了"都可能命中。Agent 要么反复误触发，要么因为匹配模糊干脆不触发。正确做法是列出具体触发短语："用户说 'review 代码'、'检查质量'、'跑 lint'"。

### 3. 步骤必须可执行

每一步都必须是 Agent **用已有工具就能完成的**。如果某一步需要 Agent 没有的工具，整个 Skill 就废了。

**反面例子**：一个 Skill 的步骤写"用 Chrome DevTools 分析页面性能"。但 Agent 没有 Chrome DevTools，它只有 headless browser。这一步对 Agent 来说不可执行，流程会在这里卡住——Agent 要么跳过，要么胡乱替代，两种结果都不可靠。

### 4. 有坑就写进去

你踩过的坑，写进 Skill。Agent 不会自己学习，每次都是从零开始读 Skill。

**反面例子**：你发现 knowledge-base Skill 用 `patch` 工具更新 index.md 会导致文件损坏，但你心想"下次注意就行"，没写进 Skill。结果下个 session，Agent 又用 `patch` 损坏了 index.md。你踩过的坑如果不写进 Skill，就等于没踩过——每个新 session Agent 都是白纸一张。

### 5. 先跑通再优化

别追求完美。先写一个能跑的版本，用几次发现问题再改。

**反面例子**：你花了一个下午写"完美的" code-review Skill，定义了 15 种检查规则、8 种输出格式、3 级严重程度。写完一跑发现步骤 2 依赖一个不存在的 MCP 工具，整个 Skill 报废。如果你先用 10 分钟写一个只有 3 步的粗版跑通，第一时间就能发现工具依赖问题，而不是在错误方向上浪费半天。

## 实战练习：拆解一个真实 Skill

拿 Hermes 的 `knowledge-base` Skill 来拆解：

```
触发：用户说"记到知识库"
  ↓
步骤1：判断页面类型（concept/entity/query）
步骤2：按模板格式 write_file
步骤3：更新 index.md（完整重写，不要用 patch）
步骤4：head -3 验证 index.md 没损坏
步骤5：追加 log.md
  ↓
踩坑：
- patch 工具会损坏 index.md → 用 write_file 完整重写
- GitHub API 限流 → 用浏览器搜索替代
```

再看 Hermes 的 `authors` Skill（写作技能）：

```
触发：用户说"写一篇文章" 或 "帮我写个 XX 主题"
  ↓
步骤1：问用户确认主题和受众（不要直接写！先对齐预期）
步骤2：根据受众选择语气（技术博客用冷静分析，公众号用口语化）
步骤3：先写大纲，让用户确认结构
步骤4：逐段写作，每段写完后停顿等待反馈
步骤5：全文写完后统一润色（去掉重复观点、统一术语）
  ↓
踩坑：
- 直接全文输出 → 用户说"重写"，浪费一个 session
- 不问受众 → 写了一篇学术风，用户说"我要的是小红书风格"
- 大纲跳过确认 → 写到一半用户说"结构不对"
```

注意 `authors` 和 `knowledge-base` 的区别：knowledge-base 是**确定性流程**（输入→格式→写入），步骤可以全自动；authors 是**交互式流程**，多个步骤需要停下来等用户确认。两种 Skill 的结构完全一样，但步骤设计思路不同。

---

你看，一个好的 Skill 就是：**触发 + 步骤 + 踩坑**。

## 你的第一个 Skill：该写什么？

从你日常最重复的动作里挑一个：

| 你常做的事 | 可以写成 Skill |
|-----------|---------------|
| 搜 GitHub 项目 | 搜索 Skill（指定怎么搜、怎么比较） |
| 整理知识库 | 知识库写入 Skill（已有） |
| 小说续写 | 续写 Skill（指定风格→读大纲→写章节） |
| 下载视频字幕 | 字幕提取 Skill |

## Skill 的进阶技巧

### 链式 Skill（一个 Skill 触发另一个 Skill）

当一个任务太复杂，拆成多个小 Skill 互相调用，而不是写一个巨无霸 Skill。

```
# deploy-and-verify Skill
步骤：
1. 执行构建命令
2. 部署到服务器
3. 触发 canary-monitor Skill  ← 链式调用
   → canary-monitor 会监控错误率、响应时间、截图对比
4. 如果 canary-monitor 报绿，标记部署成功
```

**要点**：每个被链式调用的 Skill 必须独立可用。canary-monitor 既能被 deploy Skill 调用，也能被用户直接说"监控一下生产环境"触发。

### 条件分支（根据上下文走不同路径）

用 `if/else` 让 Skill 在不同场景下走不同路径，而不是写两个几乎一样的 Skill。

```
# format-output Skill
步骤：
1. 读取目标文件
2. if 文件是 .md：
     → 用 markdown-lint 规则检查
   else if 文件是 .py：
     → 用 black + isort 格式化
   else：
     → 只做 trim 和换行标准化
3. 输出修改后的文件
```

**要点**：分支条件必须 Agent 能自动判断（文件类型、关键词、已有工具的输出结果），不要设一个 Agent 判断不了的条件。

### 参数化 Skill（从用户输入提取参数）

让 Skill 的步骤中出现 `{{占位符}}`，Agent 从用户的自然语言里提取具体值填入。

```
# translate-file Skill
触发：用户说"把 XX 翻译成 YY"

步骤：
1. 从用户消息提取参数：
   - {{source_file}}  ← "把 readme.md 翻译成日语" → readme.md
   - {{target_lang}}  ← "把 readme.md 翻译成日语" → 日语
2. 读取 {{source_file}}
3. 翻译成 {{target_lang}}，保持原格式
4. 输出到 {{source_file}}.{{target_lang}}.md
```

**要点**：参数名要直观，Agent 才能准确提取。写 `{{target_lang}}` 而不是 `{{x}}`。如果某个参数提取不到，Skill 应该主动问用户，而不是瞎猜。

## 常见错误

### 错误 1：Skill 太宽泛

**症状**：一个 Skill 试图覆盖所有场景，触发条件模糊，步骤里塞了十几个分支。

**例子**：写了一个 `helper` Skill，触发条件是"用户需要帮助"，里面包含翻译、搜索、格式化、部署等所有功能。

**解法**：拆。宁可 5 个小 Skill 各自清晰，也不要 1 个万能 Skill 没人敢用。

### 错误 2：步骤不可执行

**症状**：步骤写得像需求文档，不是操作指令。Agent 读了不知道该怎么动手。

**例子**：`"确保代码质量符合团队标准"` —— 什么叫"符合标准"？Agent 没有判断标准的能力。正确写法是 `"运行 pylint，检查分数不低于 8.0"`。

**解法**：每一步写完问自己：Agent 能用已有工具完成这一步吗？不能就改写。

### 错误 3：没有触发条件

**症状**：Skill 写得很好，但从来没被触发过——因为 Agent 不知道什么时候用它。

**例子**：一个做代码格式化的 Skill，触发条件只写了 `"用户提到代码"`。Agent 遇到用户说"这段代码有问题"时不会触发，因为匹配太模糊。

**解法**：写 3-5 个具体的触发短语，格式是"用户说了什么话"。触发条件不是描述场景，是匹配话语。

## 自测清单

写完一个 Skill，对照这 5 个问题检查：

1. **触发问题**：你随便说一句相关的用户原话，Agent 能匹配到这个 Skill 吗？试着说三句不同的话，都匹配吗？
2. **单步问题**：每一步 Agent 能用已有工具独立完成吗？有没有某一步需要"理解业务"或"做出判断"？
3. **踩坑问题**：这个 Skill 你实际跑过吗？跑的过程中卡在哪里？卡住的地方写进去了吗？
4. **范围问题**：这个 Skill 只做一件事吗？有没有"顺带也处理一下 XX"的冲动？有就忍住。
5. **精简问题**：如果删掉一步，Skill 还能跑吗？能跑就删——步骤越少越可靠。
