# Socratic Verse 教材写作规范

> 基于入门篇 5 篇核心教材（理解Agent、写Skill、工具与MCP、上下文工程、驾驭工程）的结构分析提取。

---

## 一、Frontmatter 规范

每篇教材必须以 YAML frontmatter 开头，字段如下：

### 必填字段

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `title` | string | 教材标题，与正文 H1 一致 | `理解 Agent` |
| `created` | string | 创建日期，ISO 格式 | `2026-05-19` |
| `updated` | string | 最后更新日期 | `2026-05-19` |
| `type` | string | 教材类型，固定为 `concept` | `concept` |
| `core_chain` | integer | 在核心学习路径中的序号 | `1` |
| `tags` | array[string] | 标签，至少包含一个类型标签和一个难度标签 | `[agent, concept, beginner]` |
| `confidence` | string | 置信度，可选 `high` / `medium` / `low` | `high` |
| `contested` | boolean | 是否存在有争议的内容 | `false` |
| `contradictions` | array | 已知矛盾点列表（无则留空） | `[]` |

### 必填字段格式样例

```yaml
---
title: 理解 Agent
created: 2026-05-19
updated: 2026-05-19
type: concept
core_chain: 1
tags: [agent, concept, beginner]
confidence: high
contested: false
contradictions: []
---
```

### 标签命名规范

标签数组至少包含两个标签：
1. **主题标签**：对应教材核心概念（如 `agent`、`skill`、`tool`、`context`、`harness`）
2. **分类标签**：`concept`（概念篇）或 `practice`（实践篇）
3. **难度标签**：`beginner` / `intermediate` / `advanced`

### 一致性说明

5 篇教材的 frontmatter 结构完全一致，无差异。这说明前 5 篇写于同一批次，使用了统一的模板。

## 二、内容结构模板

每篇教材遵循完全一致的六段式结构（允许省略个别段落，但顺序不变）：

### 结构模板

```
Title: # 标题（与 frontmatter title 一致）

## 开头段 — 用"一句话"或类比破题
    核心表述 + 一句通俗解释（通常 1-2 句话）

---
（分隔线）

## 核心概念展开 — 分节讲解主要内容
    ### 子节 A：概念定义 + 图示/代码示例
    ### 子节 B：对比表
    ### 子节 C：扩展讨论
    ...
    ### 最末子节：分类/类型/演变关系

---
（分隔线）

## 动手练习 — 读者可立即操作的练习
    > 目标：一句话说明练习目的与耗时
    **步骤：** 编号列表
    **思考题：** 编号列表

---
（分隔线）

## 关键原则/要点总结 — 3-5 条必须记住的核心原则
    编号列表，每条用粗体加粗标题，后跟一句话解释

---
（分隔线）

## 必读文章（可选）— 外部参考资料
    编号列表，含 URL 和一句话推荐理由

---
（分隔线）

## 关键术语速查（可选）— 术语对照表
    表格形式：术语 | 含义

---
（分隔线）

## 相关 — 内部链接
    链接列表，标注是本篇的"上一层"、"下一层"还是"相关"
```

### 各教材的段落清单

| 段落 | 理解Agent | 写Skill | 工具与MCP | 上下文工程 | 驾驭工程 |
|------|-----------|---------|-----------|------------|---------|
| H1 标题 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 开头破题 | ✅ "一句话" | ✅ "Skill 就是 SOP" | ✅ "工具=Agent的手" | ✅ "视野框"类比 | ✅ "真正的门槛"三段递进 |
| 核心概念展开 | ✅ 3节+2子节 | ✅ 5条铁律+3进阶+错误3 | ✅ 多节 | ✅ 3原则+压缩+优先级 | ✅ 6层逐层拆解 |
| 动手练习 | ✅ | ❌（无独立练习区） | ✅ 4步练习 | ✅ 3步练习 | ✅ 1步练习（较简略） |
| 要点总结 | ✅ "三件事" | ✅ "自测清单" 5问 | ❌（无独立总结区） | ❌（无独立总结区） | ❌（无独立总结区） |
| 必读文章 | ✅ | ❌ | ❌ | ❌ | ✅ "马书" |
| 术语速查 | ✅ | ❌ | ❌ | ❌ | ❌ |
| 相关链接 | ✅ | ✅ | ✅ | ✅ | ✅ |

**不一致标注：** 5 篇教材在"要点总结"、"必读文章"、"术语速查"三个段落上并不统一。这可能是教材写作中的灵活处理规则——**前三段（开头破题 → 核心概念 → 动手练习）是必选项，后四段（要点总结/必读文章/术语速查/相关链接）是可选项**。

### 分隔线规则

所有段与段之间用 `---`（三条短横线）分隔。每个段落的前后各有一条分隔线，形成清晰的视觉断点。

## 三、苏格拉底追问适用性检查清单

教材必须具备以下特征，才能被苏格拉底追问系统有效利用：

### 3.1 可推理的因果链（必备）

每篇教材必须包含至少一条明确的因果逻辑链，追问系统才能沿着它"向上追问"（why）和"向下追问"（how）。

**示例：**
- 理解Agent：Agent Loop（Thought → Action → Observation）是一条完整的因果链——每一步的"结果"决定下一步的"思考"
- 写Skill：触发条件不明确 → Agent 永不触发 / 误触发 → 任务失败
- 上下文工程：Context 满了 → 早期信息被丢弃 → Agent 失忆 → 需要压缩策略
- 驾驭工程：没有步数上限 → Agent 无限循环 → 需要 Harness 约束

**规范要求：** 每篇教材至少包含 1 条显式的"如果 X 则 Y 否则 Z"因果链。

### 3.2 对比表（推荐）

对比表是追问系统的黄金材料——读者天然想问"为什么 A 比 B 好？"。

**示例表格类型：**
| 对比类型 | 出现教材 | 追问价值 |
|----------|---------|----------|
| 概念对比 | 理解Agent：Agent vs Chatbot vs Workflow | 极高——是"分类与边界"类追问的天然来源 |
| 组件对比 | 理解Agent：核心组件表 | 高——每个组件是可独立追问的"为什么重要"节点 |
| 选项对比 | 工具与MCP：stdio vs HTTP | 高——每个维度都是追问点 |
| 策略对比 | 上下文工程：3种压缩策略对比 | 高——"什么时候用什么策略"是经典追问 |
| 层级对比 | 驾驭工程：裸Agent vs 有Harness的Agent | 高——"为什么需要更多层"是追问驱动 |

**规范要求：** 每篇教材至少包含 1 张对比表（>=2 行对比数据）。

### 3.3 反例/反面案例（推荐）

反例让读者看到"错误"而不是只有"正确"，追问系统可以问"为什么这个做法是错的？"。

**示例：**
- 写Skill：4个反面例子（万能工具箱 Skill、模糊触发条件、不可执行步骤、不写踩坑记录）
- 上下文工程：3个反面教材（3000行 API 文档塞进 CLAUDE.md、Skill 拼进 CLAUDE.md、散文风格）
- 驾驭工程：4个"没有 Harness"的问题场景

**规范要求：** 每篇教材至少包含 1 个反面案例，并用"✅正确做法 / ❌错误做法"的格式呈现。

### 3.4 数据/代码/配置示例（推荐）

具体的代码、配置、数据让追问系统可以问"如果换一个参数会怎样？"。

**示例：**
- 工具与MCP：完整的 YAML 配置片段、JavaScript 类型的工具定义、CLI 命令
- 写Skill：完整的 markdown Skill 模板、链式调用伪代码
- 上下文工程：优化前后的 CLAUDE.md 对比、压缩策略的 ASCII 图示

**规范要求：** 每篇教材至少包含 1 个可运行的代码/配置/数据示例。

### 3.5 思考题（推荐）

练习部分的思考题应设计为开放式，让追问系统可以"顺着问题继续追问"。

**规范要求：** 练习部分的思考题必须是"为什么？""如果 X 会怎样？""有什么区别？"格式的开放式问题，不接受"是/否"式问题。

### 适用性检查清单

```
□ 因果链：至少 1 条显式的"如果 X 则 Y"逻辑链
□ 对比表：至少 1 张 2+ 行对比表
□ 反例：至少 1 个 ✅/❌ 格式的反面案例
□ 示例：至少 1 个可运行的代码/配置/数据示例
□ 追问：练习部分的思考题是开放式问题
```

## 四、篇幅和概念密度标准

### 4.1 篇幅统计

| 教材 | 行数 | 字符数(约) | 概念数 | 密度(字符/概念) |
|------|------|-----------|--------|-----------------|
| 理解Agent | 221 行 | ~8.7K | ~15 | ~580 |
| 写Skill | 248 行 | ~9.7K | ~12 | ~808 |
| 工具与MCP | 582 行 | ~16.8K | ~18 | ~933 |
| 上下文工程 | 323 行 | ~15.1K | ~14 | ~1,079 |
| 驾驭工程 | 406 行 | ~26.8K | ~20 | ~1,340 |

### 4.2 标准建议

**篇幅控制：**
- **最小篇幅：** 200 行 / ~8K 字符（理解Agent为下限参考）
- **最大篇幅：** 600 行 / ~27K 字符（驾驭工程为上限参考，包含大量 ASCII图示和表格）
- **推荐篇幅：** 250-400 行 / ~10K-16K 字符

**概念密度：**
- **每个概念的间距：** 每个新概念之间至少应有 1 个例子或类比来"撑开"密度
- **密度上限：** 每 500 字符最多 1 个概念（低于 500 字符/概念时，读者信息过载）
- **密度下限：** 每 1,500 字符至少 1 个概念（高于 1,500 字符/概念时，教材太"水"）
- **推荐密度：** 每 600-1,000 字符 1 个概念

**注意：** 工具与MCP（582行）是所有教材中最长的。检查后发现其篇幅膨胀的原因是：(1) 大量 YAML 和 TypeScript 代码块（工具定义、配置示例）；(2) 5个 MCP Server 的逐一介绍（每个含表格+配置+YAML）。如果一篇教材有大量代码示例，篇幅可以远超平均线——这不视为不符合规范。

### 4.3 表格与图示的占比建议

- **表格行数：** 每 100 行正文中建议有 4-8 行表格（约 1 张表/100行）
- **ASCII图示：** 用于展示流程、架构、对比（如 Agent Loop 的箭头图、Harness 架构图）
- **代码块：** 用于展示配置示例、API 调用、工具定义

## 五、教材间衔接规范

### 5.1 核心链（core_chain）机制

每篇教材的 `core_chain` 字段定义了它在学习路径中的位置：

```
core_chain: 1  → 理解Agent
core_chain: 2  → 写Skill
core_chain: 3  → 工具与MCP
core_chain: 4  → 上下文工程
core_chain: 5  → 驾驭工程
```

### 5.2 前后引用规范

在每篇教材末尾的"相关"段落中，必须标注上/下/平行关系：

**示例（写Skill）：**
```markdown
- [[理解Agent]] — 上一层：理解 Agent 基础
- [[工具与MCP]] — 下一层：给 Skill 配工具
- projects/weread-cli — 参考：一个完整 Skill + CLI 的实现
```

**衔接标签规范：**

| 标签 | 含义 | 使用规则 |
|------|------|---------|
| 上一层 | 前置知识 | 每个教材最多 1 个 |
| 下一层 | 后续学习 | 每个教材最多 1 个 |
| 参考/相关 | 平行或扩展 | 不限数量 |

### 5.3 段落内交叉引用

教材正文中也可以直接引用其他教材：

**示例（驾驭工程中引用上下文工程）：**
```
你已经熟悉了。就是上一章 [[上下文工程]] 的内容。
```

**示例（上下文工程中引用工具与MCP）：**
```
- [[工具与MCP]] — 上一层：工具输出也占 Context
- [[驾驭工程]] — 下一层：Context 管理的工程化
```

### 5.4 引用格式

- 内部链接：使用 `[[教材标题]]` 格式 —— 注意标题与 frontmatter 的 `title` 字段完全一致
- 外部链接：使用标准 markdown 链接 `[文字](URL)`
- 项目路径：使用 `projects/project-name/` 格式引用项目目录

### 5.5 不能出现的情况

- ❌ 在 `core_chain: N` 中引用 `core_chain: N+2` 作为"上一层"（跳跃太大）
- ❌ 多个教材标为同一个 core_chain 号（除非是同一篇教材的拆分章节）
- ❌ 在教材正文中用"请参考 XX 教材"但没有 `[[ ]]` 链接

## 六、语言风格指南

### 6.1 句式特征

| 特征 | 说明 | 示例 |
|------|------|------|
| **短句优先** | 平均句长控制在 20-35 字，避免超过 50 字的长句 | ✅ "Agent 不是万能的。" / ❌ "Agent 作为一个需要依赖它所调用的工具才能完成任务的系统……" |
| **类比破题** | 每个核心概念开篇先用一个生活化的类比 | "工具 = Agent 的手"、"Context Window = 视野框"、"Agent Loop = Agent 的呼吸" |
| **提问式开头** | 新章节用问题开头，引导读者思考 | "Agent 能看到什么？"、"为什么 Agent 需要权限控制？" |
| **"一言以蔽之"结尾** | 复杂对比后用一句话总结 | "一句话区分：Chatbot 是'你说什么它回什么'……" |
| **"你"字第二人称** | 全程运用"你"，建立对话感 | "你已经在用的 Agent"、"你不用'学 Agent'" |

### 6.2 类比运用规范

类比是 Socratic Verse 教材的核心写作手法，必须遵循以下规则：

**必须使用类比的概念：**
- 所有"抽象"概念（如 Context Window、Agent Loop、MCP）
- 所有"新造词"（如 Harness、Skill）
- 所有"系统机制"（如权限控制、注入优先级）

**类比格式：**
```
**名词 = 类比对象**
一句话解释类比为什么成立
```

**示例：**
```
**Context Window（视野）** — 一次能"看到"的全部内容
| 视野范围 | 决定了 Agent 能记住多少上下文——装不下的信息等于不存在
```

**类比来源参考：**
| 抽象概念 | 已用类比 | 可替代类比 |
|----------|---------|-----------|
| Agent | — | 管家、实习生 |
| Tool | 手和脚 | 工具箱、外挂设备 |
| Memory | 便利贴 | 记事本、存档 |
| System Prompt | 员工手册 | 宪法、剧本 |
| Context Window | 视野范围 | 桌面、工作台 |
| MCP | USB 标准 | 通用插座、翻译器 |
| Skill | SOP（标准操作流程） | 食谱、操作手册 |

### 6.3 对比表写作规范

对比表是教材的"信息密度峰值"所在，必须严格按以下规范编写：

**表头格式：**
| 维度 | 项目 A | 项目 B | 项目 C |

**维度名称要求：**
- 2-5 字，清晰无歧义
- 维度与维度之间互不重叠（如不能用"速度"和"性能"同时作为两个维度——它们有重叠）
- 每个维度在 A/B/C 下都有内容，不能留空

**内容要求：**
- 每格 3-20 字，精炼
- 使用相同的句式结构（如都是"能 X"或"适合 X"）
- **加粗**标注最关键的2-3个格（如对比表中最重要的维度行）

### 6.4 代码/配置示例规范

**代码块前缀要求：**
- 在代码块前用一行文字说明"以下代码在做什么"
- 示例：
  ```
  ### 反面例子：
  （代码）
  
  ### 优化后：
  （代码）
  ```

**对比式代码必须标注✅/❌：**
```typescript
// ✅ 好：名字自解释
{ name: "read_file", description: "读取指定路径的文件内容" }

// ❌ 坏：缩写模糊
{ name: "rf", description: "read file" }
```

### 6.5 标题层级规范

| 层级 | 格式 | 用途 | 出现频率 |
|------|------|------|---------|
| H1 | `# 标题` | 教材标题（必须与 frontmatter title 一致） | 每篇 1 次 |
| H2 | `## 章节` | 主段落标题 | 每篇 4-8 个 |
| H3 | `### 子节` | 概念拆分、对比表标题 | 每个 H2 下 1-4 个 |

**禁止：** H4 及以上层级。如果某个概念需要 4 层嵌套，说明应该抽出来独立成篇。

### 6.6 特殊格式用法

| 格式 | 用法 | 示例 |
|------|------|------|
| `**粗体**` | 关键术语首次出现、table 中的关键格、要点总结的标题 | "**Agent = 能自主使用工具……**" |
| `` `行内代码` `` | 工具名、文件名、命令名、参数名 | `` `search_files`、`maxTurns` `` |
| `>` 引用 | 练习目标的用时说明、精读建议 | "> 目标：观察一次真实的 Agent Loop。5 分钟。" |
| `` ``` `` 代码块 | 配置 YAML、API 调用、伪代码、ASCII 图示 | Agent Loop 的箭头图 |
| `---` 分隔线 | 段落之间的视觉分隔 | 每段前后各一条 |

### 6.7 ASCII 图示规范

教材中的 ASCII 图示用于展示"流程"和"架构"，必须遵循以下规则：

| 规范 | 说明 |
|------|------|
| **一致的字符** | 使用 `│ ─ ┌ ┐ └ ┘ ├ ┤ ┬ ┴ ┼`（Unicode 框线字符，非 ASCII 替代品） |
| **标注节点** | 每个节点（方框）内文字不超过 10 字 |
| **标注流程方向** | 箭头 `↓ → ↑ ←` 必须与文字相邻 |
| **图下方附加文字说明** | 图示之后紧跟一个"XX图示说明"段落，否则读者看不懂 |

**示例（Agent Loop）：**
```
Thought（思考）
  → Action（行动，调用工具）
    → Observation（观察结果）
      → Thought（重新思考）
```

### 6.8 避免使用的写法

| ❌ 避免 | ✅ 替代 |
|---------|--------|
| 大段无结构的散文（>5行无标题段落） | 拆分为多个 H3 子节 |
| "此外"、"值得注意的是"等无信息量的过渡词 | 直接写内容或用分隔线 |
| 英文术语不标注中文 | 首次出现时标注：`Context Window（上下文窗口）` |
| 假设读者有前置知识而不给出路 | 每篇的"上一层"引用必须保证前置知识可达 |
| 使用"我们"代替"你" | 全程用"你"（读者视角）|

1. **写Skill 没有"动手练习"段落**——其他 4 篇都有。这是写作时的疏漏还是有意为之？如果是疏漏，建议补充。
